<task type="auto">
  <name>Task 2: Area-weighted recharge distribution with rounding residual</name>
  <files>accounting/services.py, tests/test_accounting_services.py</files>
  <action>
Fix `create_recharge_ledger_entries` (line 94) to distribute recharge volume proportionally by parcel area instead of equally.

Current (broken):
```python
per_parcel_amount = recharge_event.volume_acre_feet / Decimal(str(len(parcels)))
```

Fixed:
```python
total_area = sum(p.area_acres or Decimal("0") for p in parcels)
if total_area == 0:
    # Fallback to equal distribution if no acreage data
    per_parcel_amount = recharge_event.volume_acre_feet / Decimal(str(len(parcels)))
    # ... (existing equal distribution code)
else:
    entries = []
    distributed = Decimal("0")
    for i, parcel in enumerate(parcels):
        area = parcel.area_acres or Decimal("0")
        if i == len(parcels) - 1:
            # Last parcel gets residual to ensure exact sum
            amount = recharge_event.volume_acre_feet - distributed
        else:
            amount = (area / total_area * recharge_event.volume_acre_feet).quantize(Decimal("0.0001"))
            distributed += amount
        entries.append(ParcelLedger(..., amount_acre_feet=amount, ...))
```

Key rules:
- If ALL parcels have null/zero area_acres, fall back to equal distribution (log a warning).
- Parcels with null area_acres get 0 share (their area is treated as 0 in the weighted calc).
- Last entry gets residual: `total_volume - sum(all_other_entries)`. This is standard accounting practice to avoid rounding drift.
- Use `Decimal.quantize(Decimal("0.0001"))` for intermediate rounding (4 decimal places matches ParcelLedger's max_digits).
- Update the description string to say "area-weighted" or "equal" as appropriate.

Also update the existing test `TestCreateRechargeLedgerEntries.test_explicit_zone` — it currently asserts equal distribution (50/50 for 2 parcels). Update to create parcels with known areas (e.g., 100 acres and 300 acres) and verify the split is 25/75.

Add new tests:
- test_area_weighted_three_parcels: 3 parcels with areas 10, 20, 70 → verify 10%, 20%, 70% split
- test_area_weighted_residual: verify entries sum exactly to input volume (no rounding loss)
- test_null_area_fallback: parcels with no area_acres → equal distribution with warning

  <name>Task 4: Dashboard pro-rates allocation by account's zone coverage</name>
  <files>accounting/views.py, tests/test_views.py</files>
  <action>
Fix the dashboard view (line 72) where it sums full zone allocations for an account even when the account only owns some parcels in those zones.

Current (broken):
```python
zone_ids = ParcelZone.objects.filter(parcel_id__in=parcel_ids).values_list("zone_id", flat=True)
allocation = AllocationPlan.objects.filter(zone_id__in=zone_ids, ...).aggregate(total=Sum("allocation_acre_feet"))["total"]
```

This gives the FULL zone allocation even if the account only has 2 of 50 parcels in a zone.

Fixed: Pro-rate each zone's allocation by the account's parcel count fraction:
```python
allocation = Decimal("0")
for zone_id in zone_ids.distinct():
    zone_alloc = AllocationPlan.objects.filter(
        zone_id=zone_id,
        reporting_period=selected_period,
    ).aggregate(total=Sum("allocation_acre_feet"))["total"] or Decimal("0")
    
    total_parcels_in_zone = ParcelZone.objects.filter(zone_id=zone_id).count()
    account_parcels_in_zone = ParcelZone.objects.filter(
        zone_id=zone_id, parcel_id__in=parcel_ids
    ).count()
    
    if total_parcels_in_zone > 0:
        allocation += zone_alloc * Decimal(account_parcels_in_zone) / Decimal(total_parcels_in_zone)
```

Note: This uses parcel COUNT for pro-rating, not area. Area-weighted pro-rating would be more accurate but requires all parcels to have area_acres populated. Parcel count is a reasonable first approximation and matches how GSAs typically subdivide zones.
