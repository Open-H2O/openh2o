Fix `generate_gears_csv` by-well method (line 67) where `WellIrrigatedParcel.fraction` defaults to 1.0, causing a well irrigating 3 parcels to report 300% of actual extraction.

The bug: when building `well_parcel_map`, each WIP has `fraction=1.0` by default. The GEARS generator multiplies `abs(entry.amount_acre_feet) * fraction` per well. If a well irrigates 3 parcels and each has fraction=1.0, the well's extraction is tripled.

Fix approach — normalize fractions at query time in the generator (do NOT change the model default, which is correct for single-well parcels):

```python
# Build well→parcel map with per-well fraction normalization
raw_well_fractions = {}
for wip in WellIrrigatedParcel.objects.select_related("well").all():
    raw_well_fractions.setdefault(wip.well_id, []).append(wip)

well_parcel_map = {}  # parcel_id → [(well, normalized_fraction)]
for well_id, wips in raw_well_fractions.items():
    total_fraction = sum(w.fraction for w in wips)
    for wip in wips:
        norm_fraction = wip.fraction / total_fraction if total_fraction > 0 else Decimal("0")
        well_parcel_map.setdefault(wip.parcel_id, []).append((wip.well, norm_fraction))
```

This way, a well with 3 parcels each at fraction=1.0 gets normalized to 0.3333 each, so total = 1.0.

Also add a validation warning in `validate_report` for `gears_by_well`:
```python
# Check for wells with fractions summing != 1.0
from collections import defaultdict
well_fractions = defaultdict(Decimal)
for wip in WellIrrigatedParcel.objects.all():
    well_fractions[wip.well_id] += wip.fraction
bad_wells = [wid for wid, total in well_fractions.items() if abs(total - Decimal("1")) > Decimal("0.01")]
if bad_wells:
    warnings.append({
        "level": "warning",
        "message": f"{len(bad_wells)} well(s) have fractions not summing to 1.0 (auto-normalized for report).",
    })
```

  <name>Task 3: OpenET validation threshold accounts for temporal granularity</name>
  <files>datasync/adapters/openet.py, tests/test_openet_cache.py</files>
  <action>
Fix the `validate` method (line 130) where a hard 500mm cap rejects valid annual ET totals for irrigated Central Valley crops (which can exceed 1200mm/year).

Current (broken):
```python
elif rec["value"] > 500:
    rec["rejection_reason"] = "ET exceeds 500mm (implausible)"
```

The 500mm cap is reasonable for MONTHLY data (that's ~20 inches/month, extremely high). But OpenET can return data at different temporal granularity. For annual data, 1500mm is plausible for alfalfa or rice in the Central Valley.

Fix: make the threshold granularity-aware. Add a `temporal_resolution` parameter to `validate()`:

```python
def validate(self, records, temporal_resolution="monthly"):
    """Validate ET values. Threshold depends on temporal granularity.
    
    Thresholds:
    - daily: 15mm (max ~0.6 inches/day for peak alfalfa ET)
    - monthly: 500mm (generous cap)
    - annual: 2000mm (~79 inches, exceeds any CA crop)
    """
    THRESHOLDS = {
        "daily": 15,
        "monthly": 500,
        "annual": 2000,
    }
    max_et = THRESHOLDS.get(temporal_resolution, 500)
    
    valid = []
    rejected = []
    for rec in records:
        if rec["value"] is None:
            rec["rejection_reason"] = "null ET value"
            rejected.append(rec)
        elif rec["value"] < 0:
            rec["rejection_reason"] = "negative ET"
            rejected.append(rec)
        elif rec["value"] > max_et:
            rec["rejection_reason"] = f"ET exceeds {max_et}mm ({temporal_resolution} threshold)"
            rejected.append(rec)
        else:
            valid.append(rec)
    return valid, rejected
```

Build the missing pipeline that converts cached OpenET satellite ET data into ParcelLedger entries. Currently, the OpenET adapter fetches and caches raw mm values in `OpenETCache`, but nothing creates the negative ledger entries that represent actual water consumption.

Create management command `sync_openet_to_ledger`:

```python
# accounting/management/commands/sync_openet_to_ledger.py
"""
Convert OpenET cache records to ParcelLedger entries.

For each parcel with cached OpenET data in the given date range:
1. Read OpenETCache.et_data (list of {date, et, unit} dicts)
2. Convert mm to acre-feet: -(ET_mm / 304.8) * area_acres
3. Create ParcelLedger entry with source_type="et_estimate"
4. Skip if ledger entry already exists for that parcel+month+source_type

Formula reference:
  ET (acre-feet) = ET (mm) × area (acres) / 304.8
  304.8 = mm per foot × square feet per acre simplification
  More precisely: 1 acre-foot = 1 acre × 1 foot = 43,560 ft² × 0.3048 m/ft = 1233.48 m³
  1mm over 1 acre = 4.04686 m³, so 1 AF = 304.8 mm·acre
  Negative because ET is water CONSUMPTION (usage).
```

Key implementation details:
- Accept `--start-date`, `--end-date`, `--dry-run` arguments
- Group OpenETCache records by parcel
- For each cached record, parse the `et_data` JSON array
- Group by month (records may be daily or monthly — aggregate to monthly)
- Check for existing ParcelLedger entries with source_type="et_estimate" for that parcel+month to avoid duplicates
- Require parcel.area_acres to be non-null (skip with warning if null — Task 1 should have populated these)
- Use bulk_create with batch_size=500 (same pattern as CSV import)
- Print summary: created, skipped (existing), skipped (no area), errors

Also add a helper function in accounting/services.py:
```python
def et_mm_to_acre_feet(et_mm, area_acres):
    """Convert evapotranspiration in mm to acre-feet consumed.
    
    ET (AF) = ET (mm) × area (acres) / 304.8
    Returns negative value (ET is water consumption/usage).
    """
    return -(Decimal(str(et_mm)) / Decimal("304.8")) * area_acres
```
