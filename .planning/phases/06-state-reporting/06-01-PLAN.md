<task type="auto">
  <name>Task 1: Report generator and validator service functions</name>
  <files>reporting/generators.py, reporting/validators.py</files>
  <action>
  Create reporting/generators.py with these functions:

  **generate_gears_csv(reporting_period, method="by_well"):**
  Returns a StringIO CSV object. Two methods:
  - "by_well": Query ParcelLedger entries where source_type="meter_reading" within the reporting_period date range. Join to Well via WellIrrigatedParcel (well → parcel). Group by well + month. Columns: Well Registration ID, Well Name, Latitude, Longitude, Month (YYYY-MM), Extraction Volume (AF), Measurement Method.
  - "by_et": Query ParcelLedger entries where source_type="et_estimate" within the reporting_period. Group by parcel + month. Columns: Parcel Number, Area (acres), Month (YYYY-MM), ET Volume (AF), Measurement Method.

  **generate_calwatrs_csv(reporting_period, template_type="a1"):**
  Returns a StringIO CSV object. Query DiversionRecord filtered by reporting_period and diversion_type:
  - "a1": diversion_type="direct_use"
  - "a2": diversion_type="to_storage"
  Join through PointOfDiversion to WaterRight. Group by POD + month. Columns: Water Right ID, Holder Name, POD Name, Latitude, Longitude, Month (YYYY-MM), Volume (AF), Max Flow Rate (CFS), Diversion Type.

  **generate_email_json(reporting_period):**
  Returns a dict (JSON-serializable) with Power Automate-compatible structure:
  - agency: name, boundary info from geography.Boundary (first object)
  - period: name, start_date, end_date, is_finalized
  - summary: total_supply, total_usage, net_balance (from zone_balance or account_balance aggregation across all parcels in the period)
  - wells: list of {id, name, lat, lon, total_extraction} for the period
  - diversions: list of {water_right, pod_name, month, volume, type}
  - generated_at: ISO timestamp
  - hmac_signature: HMAC-SHA256 of the JSON payload (excluding signature field) using settings.SECRET_KEY. Use json.dumps with sort_keys=True for deterministic serialization before signing.

  Create reporting/validators.py with:

  **validate_report(reporting_period, report_type):**
  Returns a list of warning dicts [{level: "warning"|"error", message: str}]. Checks:
  - Period is finalized (warning if not)
  - Period has ledger entries (error if zero for GEARS)
  - Period has diversion records (error if zero for CalWATRS)
  - Wells have registration IDs (warning for wells missing well_registration_id, GEARS only)
  - No duplicate well-month or POD-month combinations (error)
  - Data completeness: percentage of active wells/PODs with data for each month in the period (warning if <80%)

  Use Django ORM throughout. Import from parcels.models, wells.models, surface.models, accounting.models. Return StringIO for CSVs (caller handles file writing or HTTP response). Use csv.writer with csv.QUOTE_MINIMAL.
  </action>
  <verify>
  In Django shell: from reporting.generators import generate_gears_csv, generate_calwatrs_csv, generate_email_json; from reporting.validators import validate_report. All imports succeed without errors.
  </verify>

<task type="auto">
  <name>Task 2: Management commands, reporting views, and UI</name>
  <files>reporting/management/commands/generate_report.py, reporting/management/commands/validate_report.py, reporting/views.py, reporting/urls.py, reporting/forms.py, templates/reporting/report_list.html, templates/reporting/report_generate.html, templates/reporting/report_detail.html, templates/reporting/partials/_list_results.html, templates/partials/_sidebar.html, config/urls.py</files>
  <action>
  **Management commands:**

  generate_report command:
  - Args: report_type (choices from ReportTemplate.REPORT_TYPE_CHOICES), --period (ReportingPeriod name or ID)
  - Calls the appropriate generator function
  - Writes output to stdout (CSV) or file if --output path provided
  - For email_json, writes JSON to file
  - Creates a ReportSubmission record with status="draft", stores generated_file path, validation_warnings from validate_report()

  validate_report command:
  - Args: report_type, --period
  - Calls validate_report() from validators.py
  - Prints warnings/errors with color coding (self.style.WARNING / self.style.ERROR)
  - Exits with code 1 if any errors found

  **Reporting views (reporting/views.py):**

  report_list view:
  - Lists all ReportSubmission records, newest first
  - HTMX search by report type and status filter (same pattern as parcel/well list views)
  - Shows: report type, period, status badge, generated_at, submitted_at
  - Link to generate new report and to each report detail

  report_generate view:
  - GET: Form with report_type dropdown (from active ReportTemplates) and reporting_period dropdown
  - POST: Runs validation first, displays warnings. If no errors, generates the report file, creates ReportSubmission with status="draft", redirects to detail page.
  - Store generated CSV/JSON content in ReportSubmission.generated_file as a relative path under MEDIA_ROOT/reports/ (create directory). Use format: {report_type}_{period_name}_{timestamp}.{csv|json}

  report_detail view:
  - Shows report metadata, validation warnings (if any), status
  - Download button (serves the generated file)
  - Status transition buttons:
    - Draft → "Mark as Reviewed" (POST, sets status="reviewed", adds reviewer_notes from optional textarea)
    - Reviewed → "Mark as Submitted" (POST, sets status="submitted", sets submitted_at)
  - HTMX for status transitions (swap the status section)

  **Forms (reporting/forms.py):**
  - ReportGenerateForm: report_template (ModelChoiceField), reporting_period (ModelChoiceField), with dark-mode styling matching accounting/forms.py pattern

  **URL routing (reporting/urls.py):**
  - reports/ → report_list (name="report_list")
  - reports/generate/ → report_generate (name="report_generate")
  - reports/<int:pk>/ → report_detail (name="report_detail")
  - reports/<int:pk>/download/ → report_download (name="report_download")
  - reports/<int:pk>/transition/ → report_transition (name="report_transition")

  Wire reporting/urls.py into config/urls.py under path("reporting/", include("reporting.urls")).

  **Templates:**
  Follow the exact same dark-mode card-raised pattern from accounting templates. Use VanderDev tokens. HTMX for search/filter on list, status transitions on detail.

  report_list.html: extends base.html, card-raised container, table with status badges (green=submitted, yellow=reviewed, gray=draft), HTMX search partial.

  report_generate.html: extends base.html, form card with dropdowns, validation warnings displayed as yellow/red alert boxes before confirming generation.

  report_detail.html: extends base.html, metadata card, warnings card (if any), download button, status transition buttons with HTMX.

  **Sidebar:** Add "Reports" link under a new "Reporting" section in _sidebar.html, after the "External Data" section. Icon: clipboard or document icon (use the same inline SVG pattern as other sidebar items).

  **MEDIA_ROOT:** Add MEDIA_ROOT and MEDIA_URL to config/settings/base.py if not already set. MEDIA_ROOT = BASE_DIR / "media", MEDIA_URL = "/media/". Add media URL pattern to config/urls.py for development (only when DEBUG=True).
