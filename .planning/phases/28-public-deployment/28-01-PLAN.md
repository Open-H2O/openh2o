<task type="auto">
  <name>Task 2: Create the tunnel and route openh2o.com → Butler</name>
  <action>
On Butler (once the zone is Active):
- `cloudflared tunnel login` → user opens the printed URL, authorizes openh2o.com (browser, one step).
- `cloudflared tunnel create openh2o` → credentials json in ~/.cloudflared/.
- Write ~/.cloudflared/config.yml: tunnel id, credentials-file, ingress
  `openh2o.com → http://localhost:80` (Caddy), fallback 404.
- `cloudflared tunnel route dns openh2o openh2o.com` → creates the proxied CNAME.
- `sudo cloudflared service install` → run as systemd service; enable + start.
  </action>
  <verify>`cloudflared tunnel info openh2o` shows a healthy connection; service is active.</verify>
</task>

<task type="auto">
  <name>Task 3: Caddy X-Forwarded-Proto fix</name>
  <action>
Tunnel→Caddy is plain HTTP, but the edge is HTTPS. In the Caddyfile :80 site, force the upstream
proto so Django's SECURE_SSL_REDIRECT/SECURE_PROXY_SSL_HEADER see https:
`reverse_proxy web:8000 { header_up X-Forwarded-Proto https }`. Rebuild/reload Caddy.
WHY: without it Django thinks the request is http, SECURE_SSL_REDIRECT issues a 301 to https,
the tunnel re-delivers it as http → infinite redirect.
  </action>
  <verify>curl through the tunnel returns 200/302, not a redirect loop.</verify>
</task>

<task type="auto">
  <name>Task 4: Flip to production settings</name>
  <action>

- CSRF_TRUSTED_ORIGINS=https://openh2o.com
