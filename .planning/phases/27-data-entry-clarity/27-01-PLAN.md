<objective>
Add recharge data-entry forms (events and measurements) directly on the recharge site
detail page, and make ledger source types visually distinct with color-coded pill badges.

Purpose: Today a GSA admin cannot record a recharge event from the UI — RechargeEvent and
RechargeMeasurement records can only be created via the Django admin or seed scripts. This
plan closes that gap and auto-generates the positive ParcelLedger entries that recharge
should produce. The ledger badges make the balance sheet readable at a glance (which entries
are ET estimates vs meter reads vs recharge vs budget).
Output: New recharge forms/views/URLs, HTMX inline entry forms on the site detail page, and
colored source-type pills on the ledger list.
</objective>

**Tech stack available:** Django 5.x + GeoDjango, HTMX (CDN), MapLibre GL JS, Tailwind
standalone (compiled `static/css/app.css`), pytest + factory_boy test suite (186 tests).

**Established patterns:**
- ModelForm with explicit `widgets` dict applying `form-input`/`form-select`/`form-textarea`
  classes (see `accounting/forms.py`).
- Create views: POST → `form.is_valid()` → `form.save(commit=False)`, set `created_by`,
  `save()`, then `redirect(...)`. GET renders the form (see `accounting/views.py:562`
  `ledger_create`).
- HTMX list refresh: view checks `request.headers.get("HX-Request")` and returns a
  `partials/_*.html` fragment instead of the full page (see `recharge_sites_list`).
- Badge include pattern: `{% include "recharge/partials/_status_badge.html" with status=... %}`
  emits `<span class="badge badge-pill badge-{color}">`. Badge CSS already exists in
  `static/css/app.css` (`.badge-pill` + `.badge-green/orange/red/blue/teal/grey/gold`).
- `create_recharge_ledger_entries(recharge_event, zone=None)` in `accounting/services.py:159`
  already does area-weighted distribution with rounding-residual handling. It is already
  unit-tested (Phase 22). It RAISES `ValueError` if the event's site has no `zone` FK and no
  zone is passed.

**Constraining decisions:**
- 22-01: Recharge distribution is area-weighted; do NOT reimplement — call the existing
  service function.
- 26-01: Numeric display uses `|floatformat:2`; recharge area badge colors must reuse the
  shared token classes, never new hex values.

**Constraining facts (verified in code):**
- `RechargeEvent` fields: `recharge_site, start_date, end_date(nullable), volume_acre_feet,
  water_type(nullable), source_description, notes`.
- `RechargeMeasurement` fields: `recharge_site, measurement_date(DateTimeField),
  measurement_type(choices), value, unit, notes`.
- `RechargeSite.zone` is nullable — the ledger function will raise on sites without a zone.
- `recharge/urls.py` `app_name = "recharge"`; detail route is `name="detail"`.
- `ParcelLedger.SOURCE_TYPE_CHOICES`: meter_reading, et_estimate, manual_entry, csv_import,
  surface_diversion, recharge, allocation, adjustment.
- The ledger `source_type` filter is ALREADY built (`accounting/views.py:508,523`); this plan
  does NOT touch filtering, only badge rendering.

  <files>recharge/forms.py (new), recharge/views.py, recharge/urls.py</files>
  <action>
Create `recharge/forms.py` with two ModelForms mirroring `accounting/forms.py` widget conventions:

- `RechargeEventForm` (model `RechargeEvent`): fields `start_date, end_date,
  volume_acre_feet, water_type, source_description, notes`. EXCLUDE `recharge_site` (set in
  the view from the URL pk). Widgets: dates → `forms.DateInput(attrs={"class":"form-input","type":"date"})`,
  `volume_acre_feet` → `NumberInput(attrs={"step":"0.0001","class":"form-input"})`,
  `water_type` → `Select(class="form-select")`, `source_description` →
  `TextInput(class="form-input")`, `notes` → `Textarea(rows=2, class="form-textarea")`.
- `RechargeMeasurementForm` (model `RechargeMeasurement`): fields `measurement_date,
  measurement_type, value, unit, notes`. EXCLUDE `recharge_site`. Widgets: `measurement_date`
  → `DateTimeInput(attrs={"class":"form-input","type":"datetime-local"})`, `measurement_type`
  → `Select(class="form-select")`, `value` → `NumberInput(step="0.0001", class="form-input")`,
  `unit` → `TextInput(class="form-input")`, `notes` → `Textarea(rows=2, class="form-textarea")`.

In `recharge/views.py` add two `@login_required` views:

- `recharge_event_create(request, pk)`: get site by pk. On POST, bind `RechargeEventForm`;
  if valid, `event = form.save(commit=False); event.recharge_site = site; event.save()`. Then
  attempt ledger generation:
  ```
  from accounting.services import create_recharge_ledger_entries
  try:
      created = create_recharge_ledger_entries(event)
      ledger_msg = f"Created {len(created)} ledger entr{'y' if len(created)==1 else 'ies'} across zone parcels."
  except ValueError:
      ledger_msg = "Event saved. No zone assigned to this site, so no ledger entries were generated — assign a zone to auto-distribute recharge."
  ```
  Re-query the site's events (ordered `-start_date`, `select_related('water_type')`), build
  context `{site, events, event_form: RechargeEventForm(), ledger_msg}`, and render the
  `recharge/partials/_event_history.html` fragment (HTMX target). If the form is invalid and
  it's an HX request, re-render the form partial with errors (see Task 2 for partial names).
- `recharge_measurement_create(request, pk)`: same shape for `RechargeMeasurementForm`; on
  success re-query `recent_measurements` (`-measurement_date`, `[:10]`) and render
  `recharge/partials/_measurements.html`. No ledger side effects.

Both views: on GET, just redirect to `recharge:detail` (forms live inline on the detail page,
not standalone). Use `redirect("recharge:detail", pk=pk)`.

In `recharge/urls.py` add:
```
path("<int:pk>/events/add/", views.recharge_event_create, name="event_create"),
path("<int:pk>/measurements/add/", views.recharge_measurement_create, name="measurement_create"),
```

WHY catch ValueError and not pre-check: the service is the single source of truth for the
zone rule; duplicating the check invites drift. Saving the event even when no ledger entries
result is correct — the event is real data; ledger distribution is a downstream convenience.
  </action>
  <verify>`docker compose exec web python manage.py check` passes; `python -c "from recharge.forms import RechargeEventForm, RechargeMeasurementForm"` imports clean inside the web container.</verify>
  <done>Both forms importable, both views defined, two new URL names resolve (`recharge:event_create`, `recharge:measurement_create`).</done>
</task>
