<task type="auto">
  <name>Task 2: import_ledger_csv command and diversion/recharge ledger integration</name>
  <files>accounting/management/commands/import_ledger_csv.py, surface/models.py, recharge/models.py, accounting/services.py</files>
  <action>
  **import_ledger_csv management command:**
  - Accepts CSV file path as argument. Required columns: parcel_number, effective_date, amount_acre_feet, source_type. Optional columns: water_type_code, description, transaction_date.
  - Validates: parcel_number exists in Parcel table, amount is numeric, dates parse correctly, source_type is valid choice, water_type_code matches WaterType.code if provided.
  - Creates ParcelLedger entries in bulk (batch of 500). Reports: created count, skipped count, error rows with line numbers.
  - Options: --dry-run (validate only, don't create), --reporting-period (name to assign), --delimiter (default comma).
  - Follow the same style as import_parcels and import_wells commands.

  **Diversion → Ledger integration (in accounting/services.py):**
  - create_diversion_ledger_entry(diversion_record): Given a DiversionRecord, creates a negative ParcelLedger entry on the first parcel linked to the water right's holder. Amount = -diversion_record.volume_acre_feet. Source type = "surface_diversion". Links to the reporting_period from the diversion record.
  - This is a utility function called manually or from a management command, NOT automatic on DiversionRecord save (keep it explicit, not magic).

  **Recharge → Ledger integration (in accounting/services.py):**
  - create_recharge_ledger_entries(recharge_event): Given a RechargeEvent, creates positive ParcelLedger entries distributed across parcels in the recharge site's zone. Amount = +recharge_event.volume_acre_feet / parcel_count. Source type = "recharge".
  - Also a utility function, not an automatic signal.

  NOTE: These integration functions are helpers. The diversion/recharge views remain read-only (Phase 3 decision). A future management command or admin action could call them in batch.
  </action>

<task type="auto">
  <name>Task 3: Balance calculation engine and account detail with balances</name>
  <files>accounting/services.py, accounting/views.py, templates/accounting/account_detail.html</files>
  <action>
  **Balance calculation utilities (in accounting/services.py):**
  - parcel_balance(parcel, reporting_period=None): Sum of ParcelLedger.amount_acre_feet for a parcel, optionally filtered by reporting_period. Returns Decimal. Uses Django ORM aggregate(Sum).
  - account_balance(water_account, reporting_period=None): Sum of parcel_balance for all parcels linked to the account (via WaterAccountParcel). Returns dict: {total, supply (sum of positives), usage (sum of negatives abs), net}.
  - zone_balance(zone, reporting_period=None): Sum of parcel_balance for all parcels in the zone (via ParcelZone). Returns same dict structure as account_balance.
  - All functions handle empty querysets (return Decimal('0') or zeroed dict).

  **Enhanced account detail view:**
  - Update account_detail to call account_balance() and display: total supply (green), total usage (red), net balance (green if positive, red if negative).
  - Show per-parcel balance breakdown table: parcel_number, supply, usage, net for each assigned parcel.
  - Add reporting_period selector (dropdown) to filter balances by period. Default to most recent non-finalized period if one exists, otherwise show all-time.
  - Use HTMX to reload balance section when period changes (hx-get with period param, returns partial).

  **Templates:**
  - Balance display uses card-raised styling. Supply in green (--color-forest-teal or similar), usage in red (--color-furnace-orange), net uses conditional color.
  - Numbers formatted with tabular-nums font-variant for alignment.
