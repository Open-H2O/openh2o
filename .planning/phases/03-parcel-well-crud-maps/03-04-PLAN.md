<task type="auto">
  <name>Task 1: Import management commands for parcels and wells</name>
  <files>parcels/management/commands/import_parcels.py, wells/management/commands/import_wells.py</files>
  <action>
  import_parcels command:
  - Arguments: file_path (positional), --format (geojson|shapefile, auto-detect from extension), --parcel-number-field (default: APN), --owner-field (default: OWNER), --dry-run flag
  - Uses django.contrib.gis.gdal.DataSource to read both GeoJSON and Shapefile
  - For each feature:
    1. Extract parcel_number from the configured field name
    2. Check for duplicate in Parcel table (by parcel_number)
    3. If new: create ParcelStaging record with raw_data=feature properties, geometry=feature geometry (transform to SRID 4326 if needed), status='pending'
    4. If duplicate: create ParcelStaging with status='duplicate'
  - After staging: promote all 'pending' ParcelStaging records to Parcel table (create Parcel, update ParcelStaging status to 'imported', set imported_at)
  - Handle geometry type conversion: if input is Polygon, wrap in MultiPolygon
  - Output: summary line "Imported X parcels, Y duplicates skipped, Z errors"
  - --dry-run: stage but don't promote, report what would happen

  import_wells command:
  - Arguments: file_path (positional), --format (csv|shapefile, auto-detect from extension), --name-field (default: WELL_NAME), --lat-field (default: LATITUDE), --lon-field (default: LONGITUDE), --reg-id-field (default: WELL_REG_ID), --dry-run flag
  - For CSV: use csv.DictReader, create Point from lat/lon fields
  - For Shapefile: use DataSource, extract Point geometry
  - Duplicate detection: by well_registration_id if provided
  - Create Well records directly (no staging table for wells)
  - Handle: missing coordinates (skip with warning), missing names (use reg ID or "Well N")
  - Output: summary line "Imported X wells, Y duplicates skipped, Z errors"

  Both commands: use self.stdout.write with self.style.SUCCESS/WARNING/ERROR for colored output. Wrap in transaction.atomic() for the promotion step.
  </action>
  <verify>Create a test GeoJSON file with 3 parcels. Run: docker compose exec web python manage.py import_parcels /path/to/test.geojson --parcel-number-field=APN. Verify 3 parcels created. Run again -- 0 new (duplicates). Create a test CSV with 2 wells. Run import_wells. Verify 2 wells created.</verify>
  <done>import_parcels handles GeoJSON and Shapefile, stages then promotes, detects duplicates. import_wells handles CSV and Shapefile, creates wells directly. Both support --dry-run.</done>
</task>

  <name>Task 2: Surface water and recharge views</name>
  <files>surface/views.py, surface/urls.py, templates/surface/water_rights_list.html, templates/surface/water_right_detail.html, templates/surface/partials/_list_results.html, recharge/views.py, recharge/urls.py, templates/recharge/sites_list.html, templates/recharge/site_detail.html, templates/recharge/partials/_list_results.html</files>
  <action>
  Surface water views (follow same patterns as parcel/well views from Plan 03):

  water_rights_list:
  - GET /surface/ with HTMX search (right_id__icontains OR holder_name__icontains) and status filter
  - Table columns: Right ID (link to detail), Type, Holder, Priority Date, Face Value (AF), Status
  - Same partial-response pattern for HTMX

  water_right_detail:
  - GET /surface/<pk>/
  - Sections: Right Info, Points of Diversion (list with map showing POD locations), Recent Diversion Records (last 12 months), Curtailment Status (if any active curtailment applies based on priority_date)
  - Embedded map showing all PODs for this water right

  Recharge views:

  recharge_sites_list:
  - GET /recharge/ with HTMX search (name__icontains) and site_type filter
  - Table columns: Name (link to detail), Type, Capacity (AF), Status, Operator
  - Same partial-response pattern

  recharge_site_detail:
  - GET /recharge/<pk>/
  - Sections: Site Info, Event History (chronological list of RechargeEvents with volume and water type), Recent Measurements (last 10 RechargeMeasurements)
  - Embedded map showing site location

  All views: @login_required, VanderDev dark-mode styling, consistent with parcel/well view patterns.
  No inline editing on these views (water rights and recharge data typically comes from external sources, not manual entry).
  </action>
