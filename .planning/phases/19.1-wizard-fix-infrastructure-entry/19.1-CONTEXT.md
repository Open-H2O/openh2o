# Phase 19.1: Wizard Fix & Infrastructure Entry Forms - Context

**Gathered:** 2026-05-25
**Status:** Ready for planning

<vision>
## How This Should Work

First, fix the broken setup wizard -- the confirm step currently renders an empty tile. That's the prerequisite before adding anything new.

Then, build a unified infrastructure entry system where users can plot or upload their physical water infrastructure: points of surface water diversion, groundwater extraction wells, storage ponds/tanks, and points of re-diversion or return to waterway. All four types share one form with a type selector, and the fields adjust per type (well gets depth fields, diversion gets intake type, etc.).

Two equal paths for getting features into the system:
1. **Map plotting** -- click on the map to place a point or draw a polygon. The map interaction needs to feel smooth and intuitive. This is the most important thing to nail.
2. **File upload** -- drag in GeoJSON, Shapefile, or KML and see the results on the map.

Each infrastructure feature gets linked to a parcel (place of use) through three options: pick from an existing parcel already in the system, select a parcel by clicking on the parcel map layer, or draw a new polygon on the spot.

The forms live in two places: as a step in the existing /setup/ wizard for initial onboarding, and as a standalone page under the sidebar for ongoing additions after setup is complete.

</vision>

<essential>
## What Must Be Nailed

- **Map interaction quality** -- plotting points and drawing polygons on MapLibre needs to feel responsive and natural. If the map UX is clunky, the whole feature fails.
- **Wizard fix** -- the broken confirm step must work before anything else gets built on top of the wizard.
- **Parcel linkage flexibility** -- three paths (pick existing, select from map, draw new) so users aren't blocked regardless of their starting point.

</essential>

<boundaries>
## What's Out of Scope

- No water rights linkage -- infrastructure connects to parcels, not water rights. Water rights association is a separate concern.
- Reporting metadata (GEARS well construction details, CalWATRS IDs) -- deferred to Phase 19.2.
- Telemetry source selection (CDEC/USGS/CIMIS/etc. enable/disable) -- deferred to Phase 19.2.
- Accounting integration -- these forms create spatial features; wiring them into water balance calculations is not part of this phase.

</boundaries>

<specifics>
## Specific Ideas

- One unified "Add Infrastructure" form with a type dropdown that adapts visible fields per infrastructure type.
- Both map-first and upload-first workflows presented with equal visual weight -- no "primary" path.
- Supported upload formats: GeoJSON, Shapefile (zipped), KML.
- Infrastructure types: Surface Water Diversion, Groundwater Well, Storage (pond/tank), Re-diversion/Return.
- The existing setup wizard at /setup/ gains an infrastructure step; the same UI is also accessible standalone from the sidebar navigation.

</specifics>

<notes>
## Additional Context

Phase 19 (just completed) built the monitoring dashboard and setup wizard. The wizard's confirm step broke -- likely a rendering issue in the HTMX partial. The auto_populate engine (Phase 17-18) handles automated data loading; this phase adds manual infrastructure entry for features that aren't in public datasets.

The existing data model already has relevant tables: PointOfDiversionParcel, WellIrrigatedParcel, and related models from Phase 2-3. This phase adds the user-facing entry forms and map interactions, not new data models.

Phase 19.2 will cover the configuration side: GEARS/CalWATRS metadata fields and telemetry source selection.

</notes>
