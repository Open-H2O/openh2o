<objective>
Make the Use Ledger (and the table pattern it sets) usable at real-district scale —
thousands of parcels, hundreds of thousands of rows — by layering three navigation
tools onto the EXISTING `ledger_list` view and its two templates: sortable columns
with sticky header + search bar, a Zone facet alongside the existing filters, and a
page-size selector (25/100/500) with jump-to-page.

Purpose: Today the ledger is fine at demo size but would be an unnavigable wall of
rows for a real district. The accounting data is already correct; this phase makes it
*drivable*. Whatever pattern lands here is the template every other large table copies.

Output: An upgraded `accounting:ledger_list` — sortable/sticky/faceted/paged — with
view-layer tests for the new query handling, verified live on openh2o.com.
</objective>

**Established patterns (already in `ledger_list`):**
- Live HTMX filtering: each control has `hx-get` + `hx-target="#ledger-results"` +
  `hx-trigger` (change / keyup delay) + an `hx-include` enumerating the OTHER controls.
- Newest-first default: `.order_by("-effective_date", "-created_at")`.
- Bare-landing period auto-default (ISS-022): when no `period` key and no other
  filters, default to the most recent period that has calculated rows. **Preserve this.**
- HTMX requests return the partial; full requests return the page.

**Constraining decisions:**
- **Parcel facet = the existing search box** (Brent, 2026-06-05). It already filters
  `parcel__parcel_number__icontains`. Do NOT add a thousands-row parcel `<select>` —
  that would be the at-scale problem this phase exists to kill. The new facet to add
  is **Zone**. Relabel/clarify the search box as the parcel filter.
- Default page size = **100** (CONTEXT options are 25/100/500; 100 is the at-scale
  middle, replacing the current hardcoded 50).
- Out of scope this phase: column show/hide, CSV export changes, ISS-058 (GEE re-fetch).

**Sticky-CSS gotcha (verified):** the scroll container is `.app-content`
(`overflow-y: auto`, app.css:115). `position: sticky; top: 0` sticks relative to it.
BUT `_ledger_list_results.html` wraps the `<table>` in
`<div class="card-raised" style="...overflow: hidden;">` — an `overflow:hidden`
ancestor CLIPS a sticky `<thead>`. Task 3 must remove that `overflow:hidden` for the
header to pin. Filter bar and header row both sticky → stack their `top` offsets so
they don't overlap.

1. **Sortable columns.** Read `sort = request.GET.get("sort","")` and
   `direction = request.GET.get("dir","desc")`. Map sort keys to ORM fields via an
   explicit WHITELIST dict (never interpolate user input into order_by):
     SORTABLE = {
       "date": "effective_date",
       "parcel": "parcel__parcel_number",
       "amount": "amount_acre_feet",
       "source": "source_type",
       "water_type": "water_type__name",
     }

   direction == "desc", then a stable secondary tiebreak `"-created_at"`. If `sort`
   is absent/invalid: keep the current default `("-effective_date", "-created_at")`
   (newest-first — the CONTEXT default). WHY a whitelist: an unbounded order_by from
   a GET param is an injection / 500 vector; fail closed to the default.

2. **Zone facet.** Read `zone_id = request.GET.get("zone","").strip()`. When set,
   `queryset = queryset.filter(parcel__parcel_zones__zone_id=zone_id)`. The
   parcel↔zone join is many-to-many through ParcelZone, so a parcel in N zones could
   duplicate rows — append `.distinct()` to the queryset when zone filtering is
   active (or unconditionally after filters, before pagination). Add
   `zones = Zone.objects.order_by("name")` and pass to context as `zones`; pass the
   selected `zone_id` back for the <select>. Import Zone from geography.models.

3. **Page size + jump-to-page.** Read `page_size` from GET; validate against
   `PAGE_SIZES = (25, 100, 500)`, default 100 (replaces the hardcoded `Paginator(queryset, 50)`).
   Coerce safely: `try: page_size = int(...); page_size = page_size if page_size in PAGE_SIZES else 100; except: 100`.
   `Paginator.get_page` already clamps an out-of-range/non-int page number, so
   jump-to-page needs no extra guard — just keep using `request.GET.get("page", 1)`.

4. **Context additions:** add `sort`, `direction`, `zone_id`, `zones`, `page_size`,
   and `PAGE_SIZES` (pass the tuple as `page_sizes`) so the templates can render the
   sort indicators, Zone <select>, and page-size <select>. Keep every existing
   context key.

**Tests** (tests/test_ledger_navigation.py — runs on Butler, settings pinned by make test):
   - `?sort=amount&dir=asc` returns rows ascending by amount; `dir=desc` descending.
   - `?sort=bogus` falls back to newest-first (no 500, first row is most recent date).
   - `?page_size=500` paginates at 500; `?page_size=9999` falls back to 100.
   - `?zone=<id>` returns only rows whose parcel is in that zone, no duplicate rows
     (assert count == distinct count) — build a parcel in 2 zones in the fixture.
   - Existing filters still compose (e.g. `?zone=<id>&source_type=calculated`).
Use factory_boy factories already in tests/ (ParcelLedgerFactory / ParcelFactory /
ZoneFactory if present; check tests/factories or conftest and reuse — do not invent).
