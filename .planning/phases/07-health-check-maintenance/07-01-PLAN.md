<task type="auto">
  <name>Task 1: Health check functions and run_health_checks command</name>
  <files>health/checks.py, health/management/__init__.py, health/management/commands/__init__.py, health/management/commands/run_health_checks.py</files>
  <action>
  Create health/checks.py with 8 check functions, each returning a dict with keys: category, status (green/yellow/red), message, details.

  1. **check_database()**: Test DB connection via django.db.connection.ensure_connection(). Count rows in key tables (Parcel, Well, ParcelLedger, WaterAccount). Green if connected, red if connection fails.

  2. **check_disk()**: Use shutil.disk_usage() on BASE_DIR and MEDIA_ROOT. Yellow if >80% used, red if >90%. Include free/total in details.

  3. **check_sync_freshness()**: Query DataSyncLog for each active DataSource. Yellow if last successful sync >48 hours ago, red if >7 days or never synced. Include per-source breakdown in details dict.

  4. **check_ledger_integrity()**: Query ParcelLedger entries where parcel_id points to a deleted or nonexistent Parcel (using left join / exclude). Check for entries with zero amount. Green if none found, yellow if zero-amount entries exist, red if orphan entries exist.

  5. **check_orphans()**: Count parcels not assigned to any WaterAccount (no WaterAccountParcel). Count wells with no associated MonitoredStation. Report counts. Yellow if any orphans, green if none.

  6. **check_ssl()**: Use Python ssl module to connect to settings.ALLOWED_HOSTS[0] (or settings.SITE_DOMAIN if defined) on port 443. Check cert expiry. Green if >30 days, yellow if <30 days, red if <7 days or connection fails. Wrap in try/except and return yellow with "SSL check unavailable (development mode)" if ALLOWED_HOSTS is empty or contains only localhost/*.

  7. **check_docker()**: From inside the container, verify DB is reachable (covered partially by check_database). Additionally check that the expected database name matches settings.DATABASES['default']['NAME']. This is a lightweight container-awareness check. Green if DB name matches expected, red if mismatch.

  8. **check_migrations()**: Use django.core.management.call_command('showmigrations', '--plan') to capture output. Parse for unapplied migrations (lines without [X]). Green if all applied, red if any unapplied.

  Create a run_all_checks() function that calls all 8 and returns a list of result dicts.

  Create health/management/commands/run_health_checks.py:
  - Call run_all_checks()
  - Save each result as a HealthCheckResult row (bulk_create)
  - Print a formatted table to stdout with colored status indicators (self.style.SUCCESS/WARNING/ERROR)
  - Add --json flag to output JSON instead of table
  - Add --category flag to run a single category only

  Create the management directory __init__.py files.

  2. **health_api(request)**: Return JsonResponse with latest results per category. Format: {"status": "healthy|degraded|unhealthy", "checks": [{"category": "...", "status": "...", "message": "...", "checked_at": "..."}], "checked_at": "..."}. Return HTTP 200 for healthy/degraded, 503 for unhealthy.

<task type="auto">
  <name>Task 3: prune_old_data management command</name>
  <files>health/management/commands/prune_old_data.py</files>
  <action>
  Create prune_old_data management command that cleans up old data:

  1. **DataRecordStaging** with status="published" and created_at older than 90 days -- these have already been published to production tables
  2. **HealthCheckResult** older than 365 days -- only latest per category matters for dashboard
  3. **DataSyncLog** older than 365 days -- keep recent history for troubleshooting

  Flags:
  - --dry-run: Print what would be deleted without deleting (default behavior, require --confirm to actually delete)
  - --confirm: Actually perform the deletion
  - --staging-days N: Override 90-day threshold for staging data (default 90)
  - --health-days N: Override 365-day threshold for health results (default 365)
  - --sync-days N: Override 365-day threshold for sync logs (default 365)

  Print counts before and after deletion. Use Django's queryset.delete() which handles cascades.
  Do NOT delete DataRecordStaging records that are not status="published" (pending/failed records need review).
  </action>
  <verify>AST parse succeeds. --dry-run mode shows counts without deleting.</verify>
  <done>prune_old_data command with --dry-run default, --confirm for actual deletion, configurable day thresholds for each data type</done>
