## The reusable shell (built, Phase 1)

Follow this recipe for every remaining screen — Wells (`wells/`) is the worked example:
- **`static/js/map-core.js` → `OH2O.detailPaneMap({card,host,dataEl,source,key,popup})`** —
  the build-once / re-point persistent detail map; auto-handles point AND polygon
  geometry (`_walkCoords`/`_geojsonBounds`/`_frameBounds`). Keyed registry
  `OH2O._detailMaps[host]` so it's reused, never rebuilt (kills the tile-flash).
- **`core/workspace.py` → `list_response` / `detail_response`** — collapse the
  `HX-Request` fragment-vs-page branch.
- **`templates/workspace.html`** — reusable two-pane shell. Standard IDs: `#results`,
  `#detail-map-card`, `#detail-map`, `#detail-body`; delegated `[data-row]` highlight.
  A screen extends it and fills blocks `master_toolbar` / `master_results` /
  `detail_body`. The per-screen `_detail_pane.html` emits `{{ geojson|json_script:"detail-geojson-data" }}`
  + a guarded `OH2O.detailPaneMap(...)` call (poll on `window.OH2O` so it survives
  full-page loads where the body renders before the map scripts).
- The standalone `detail.html` reuses the same `_detail_pane.html` fragment so the
  page and the in-pane view never drift; it always loads the map toolkit.

## What 2.0 is

Replace the old navigate-away list/detail flow (click a row → full page reload to a
separate detail page) with a **master-detail workspace**: the list stays pinned on
the left, clicking a row swaps the detail in on the right via HTMX with no reload,
map and scroll position held steady. Use Areas is the reference implementation.

The native Mac app and the web app already **share design tokens** — the gap is
STRUCTURE, not styling. The aesthetics pass (Phase 5) is the styling finish on top
of the structural work, deliberately last so it isn't redone as screens move.

## The reference pattern (built, on `feature/master-detail-use-areas`)

- `_parcel_detail_context()` is shared by all three render paths.
- `parcel_detail` returns the `_detail_pane` fragment on `HX-Request`, else the full page.
- `parcels_list` pre-loads `?selected=<pk>`.
- **Persistent map:** `#parcel-map` lives in the pane *shell* (list.html + detail.html);
  rows use `hx-target="#detail-body"` (swap the body only); the pane script builds the
  map ONCE (`window._parcelPaneMap`) then `setData` + `resize` + `fitBounds` per switch.
  NEVER rebuild the map per swap — that caused the tile-flash.
- **CSS:** `.app-main:has(.workspace-split){height:100dvh;overflow:hidden}` pins the
  viewport; panes are `height:100%` and scroll internally; `scrollbar-gutter:stable`.

## Cross-cutting (every phase)
- Verify each screen via chrome-devtools over Tailscale on staging (NOT the
  Cloudflare URL), computed styles + screenshot, before declaring done.
- **Verify with SHORT detail content, not just record-heavy PODs.** The v1.17
  footer-void bug only showed when the detail pane was short (panes collapsed
  instead of filling); a tall pane hid it. Always test the emptiest record.
- **Bump `app.css ?v=N` in `base.html` on ANY app.css change.** Staging serves
  static unhashed (DEBUG), so the `?v=` stamp is the ONLY cache key there —
  forget it and the browser serves stale CSS and the "fix" looks like it failed
  (cost a multi-round chase in the v1.17 session). Prod content-hashes anyway.
- Keep the full test suite green (`docker compose exec -T web python -m pytest --ds=config.settings.local`).
- Responsive: define how the split panes collapse on tablet/mobile.
- Nav + breadcrumb consistency as screens convert.

## Shell hardening (v1.17)
- **Panes fill via flex-stretch, NOT a `height:100%` chain.** `.app-content` (in
  a workspace) is `display:flex; flex-direction:column`; `.workspace-split` is
  `flex:1 1 auto; min-height:0`. A `height:100%` percentage only resolves against
  a *definite* ancestor height, and through a flex-grown `.app-content` that
  resolution is engine-fragile — some engines collapse the panes to content
  height (footer rode up, black void). Flex stretch fills identically everywhere.
- **Feedback button floats on workspace pages.** `.app-main:has(.workspace-split)
  > .oh2o-fb-btn { position:fixed }`. It's normally a sticky last-child that nests
  into the footer on scroll; on a pinned (100dvh, no-scroll) workspace that in-flow
  box pushed the footer below the fold. No scroll to ride here, so float it.
