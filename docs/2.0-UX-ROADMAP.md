# OpenH2O 2.0 — Master-Detail UX Re-Engineering

**Status:** In progress. Use Areas (parcels) converted and verified on staging; the
rest of the app is pre-2.0. **Scope decision (2026-06-21, Brent):** full overhaul —
*all* list-detail screens converted, *and* the Apple aesthetics pass included. 2.0
ships only when the app both works the new way and looks like the native Mac app.

## What 2.0 is

Replace the old navigate-away list/detail flow (click a row → full page reload to a
separate detail page) with a **master-detail workspace**: the list stays pinned on
the left, clicking a row swaps the detail in on the right via HTMX with no reload,
map and scroll position held steady. Use Areas is the reference implementation.

The native Mac app and the web app already **share design tokens** — the gap is
STRUCTURE, not styling. The aesthetics pass (Phase 5) is the styling finish on top
of the structural work, deliberately last so it isn't redone as screens move.

## The reference pattern (built, on `feature/master-detail-use-areas`)

- `_parcel_detail_context()` is shared by all three render paths.
- `parcel_detail` returns the `_detail_pane` fragment on `HX-Request`, else the full page.
- `parcels_list` pre-loads `?selected=<pk>`.
- **Persistent map:** `#parcel-map` lives in the pane *shell* (list.html + detail.html);
  rows use `hx-target="#detail-body"` (swap the body only); the pane script builds the
  map ONCE (`window._parcelPaneMap`) then `setData` + `resize` + `fitBounds` per switch.
  NEVER rebuild the map per swap — that caused the tile-flash.
- **CSS:** `.app-main:has(.workspace-split){height:100dvh;overflow:hidden}` pins the
  viewport; panes are `height:100%` and scroll internally; `scrollbar-gutter:stable`.

## Phases

### Phase 0 — Ship Use Areas to prod
Done + verified on staging, flash-free, 862 tests pass. Awaiting Brent's visual OK,
then `ssh butler 'cd ~/openh2o && make deploy'` and tag the release. Gives us the
pattern live and a real reference. **(Blocked only on go-ahead.)**

### Phase 1 — Extract the reusable workspace shell *(the leverage move)*
Lift the master-detail pattern out of the parcels templates into a generic,
fill-in-the-blanks shell (shared `workspace-split` layout + persistent-map JS +
detail-context helper convention) so each later conversion is a casting from the
mold, not a from-scratch rebuild. Do this BEFORE Wells.

### Phase 2 — Spatial conversions (list + live map + detail)
The map-first screens, highest visible payoff. One at a time, each verified on staging.
1. **Wells** (`/wells/`)
2. **Surface Diversions / PODs** (`/surface/`)
3. **Water Rights** (`/surface/rights/`) — maps the PODs it serves
4. **Recharge Areas** (`/recharge/`)
5. **Monitoring Stations** (`/datasync/stations/`)
6. **Zones** (`/map/zones/`, admin)

### Phase 3 — Records & ledger conversions (list + detail, usually no map)
1. **Use Ledger** (`/accounting/ledger/`)
2. **Water Accounts** (`/accounting/accounts/`)
3. **Reports** (`/reporting/reports/`)
4. **Water Years / Periods** (`/accounting/reporting-periods/`, admin)
5. **Allocations** (`/accounting/allocations/`, admin) — list-heavy; lightest touch

### Phase 4 — Singleton screen polish (not split-pane; bring up to 2.0 standard)
Accounting Dashboard · full-screen Map · Setup & Import wizards · Methodology &
Delivery Settings · the "How was this calculated?" calculation-run audit trail ·
report-generation and CalWATRS worksheet forms · Home / About / Profile / Users /
Help pages.

### Phase 5 — Apple aesthetics pass (global)
Type, spacing, depth, and motion refinement across every screen so the web app
reads like the native Mac app. Last, by design — structure stops moving first.
