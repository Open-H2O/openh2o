# OpenH2O 2.0 — WCAG 2.1 AA / Section 508 Accessibility Audit

**Date:** 2026-06-23
**Branch audited:** `feature/master-detail-stations` (staging, app.css v60)
**Standard:** WCAG 2.1 Level AA, which Section 508 (Revised, 2017) incorporates
by reference. This is the bar a California state government tool must meet.
**Status:** Findings only — nothing here is fixed yet. This is the deferred
Tier-3 compliance pass from the UX Pattern Spec (E3 shipped the skip-link quick
win; this is the formal audit).

## Method

Four layers, because no single tool catches everything:

1. **Hard contrast math** — every text/surface colour pair from `tokens.css`
   run through the WCAG relative-luminance formula (not eyeballed).
2. **Automated scan** — Lighthouse accessibility audit (axe-core engine) on the
   Use Ledger (a representative dense page: table, links, badges, forms, modal).
   Score: **88 / 100**, 220 contrast elements flagged.
3. **Semantic + keyboard testing** — scripted DOM checks (landmarks, headings,
   labels, alt text, focus styles) plus a real keyboard test of the custom
   master-detail row pattern.
4. **Source tracing** — every finding traced to the file/line that causes it, so
   remediation is a known edit, not a hunt.

## Findings

### 🔴 F1 — Master-detail rows are not keyboard-operable
**WCAG 2.1.1 Keyboard (Level A) · 508 §502.2.1 · CRITICAL**

The list rows on **Wells, Use Areas, and Water Accounts** are
`<div role="link" tabindex="0">` with an `hx-get` and **no `hx-trigger`**, so
HTMX falls back to its default trigger: mouse **`click`**. A `<div>` does not
synthesize a click from the keyboard the way a real `<button>`/`<a>` does.

**Verified live:** focused the first well row (it takes focus fine), pressed
**Enter** — the detail pane did not change. The row is reachable by keyboard but
cannot be activated. A keyboard-only or screen-reader user cannot open any
record on the three master-detail screens. This is the single most serious
finding: 2.1.1 is the most basic 508 requirement.

- **Where:** `templates/wells/partials/_list_results.html`,
  `templates/parcels/partials/_list_results.html` (Use Areas),
  `templates/accounting/partials/_account_list_results.html` — the `.master-row`
  element.
- **Fix (low effort):** add keyboard triggers to the row, e.g.
  `hx-trigger="click, keyup[key=='Enter'], keyup[key==' ']"` and a tiny handler
  to `preventDefault` on Space so it doesn't scroll. Better still, audit whether
  the row can be a real `<a href="...?selected=pk">` (it already sets
  `hx-push-url` to that URL) — a native link is keyboard-correct for free and
  degrades without JS.

### 🟠 F2 — Body-text contrast falls below AA
**WCAG 1.4.3 Contrast (Minimum) (Level AA) · 508 §503 · SERIOUS**

Lighthouse flagged **220 elements**; the contrast math pins down exactly which
tokens and where. AA needs **4.5:1** for normal text, **3:1** for large text
(≥18.66px bold / ≥24px) and UI components.

| Foreground | On surface | Ratio | Verdict |
|---|---|---|---|
| `--color-blue` #1B7FAF (`.data-table-link`, `.text-blue`, `.badge-blue`) | card #080b10 | **4.41:1** | ✗ normal text |
| `--color-blue` on inset #12151b | inset | **4.09:1** | ✗ |
| `--color-blue` on elevated #242a33 | elevated | **3.23:1** | ✗ |
| `--color-text-tertiary` #6b7f94 on inset | inset | **4.43:1** | ✗ (the "How the ledger works" explainer box) |
| `--color-text-tertiary` on elevated | elevated | **3.50:1** | ✗ |
| `.text-supply` green (flagged by axe) | card | <4.5 | ✗ |

The worst offender by reach is **`.data-table-link`** — the blue identifier link
in *every* data-table row (every parcel/well/record). It is ~14px, weight 600,
at 4.41:1 on a card.

- **What passes and points to the fix:** the **brighter** blue
  `--color-blue-bright` #3DB4E0 scores **8.26:1** on a card, and the darker blue
  is fine for borders, fills, icons, and large headings (all ≥3:1).
- **Fix (low–medium; intersects Phase F):** route *text-sized* blue (links,
  badge text, `.text-blue`) to `--color-blue-bright`, and nudge
  `--color-text-tertiary` a few points lighter (a value clearing 4.5:1 on the
  `elevated` surface covers every surface). Recheck `.text-supply` after. Because
  this touches core theme tokens, it is the one finding worth folding into the
  Phase F colour pass rather than patching blindly.

### 🟠 F3 — Every page's `<h1>` is the app name, not the page
**WCAG 1.3.1 Info & Relationships (A) + 2.4.6 Headings & Labels (AA) · SERIOUS**

The header's `<h1>` is `{% block page_title %}` inside
`templates/partials/_header.html`, but that partial is pulled in with
**`{% include %}`** (base.html:96). Django block overrides only flow through
`{% extends %}`, **not `{% include %}`** — so the `{% block page_title %}MyPage{% endblock %}`
that **57 page templates** define is dead code for the `<h1>`. The heading
always renders the default, **"Open Water Accounting Platform"**, on every page.

**Consequence:** a screen-reader user navigating by headings hears the same app
name on every page; the actual subject ("Use Ledger") lives only in a breadcrumb
`<span>`. The heading tree on the ledger is literally `H1 (app name) → H3
(feedback modal)` — no page-level heading and a skipped level.

This is both an accessibility finding **and a latent template bug** (a
deliberately-overridden block silently doing nothing).

- **Fix (low–medium):** make the page subject a real `<h1>`. Either move the
  title `<h1>` out of the included partial into the inheritance chain, or pass it
  into the include explicitly (`{% include "partials/_header.html" with page_subject=... %}`),
  and demote the app-name to non-heading branding. Verify no level is skipped.

### ✅ F4 — Links distinguishable by colour alone — RESOLVED
**WCAG 1.4.1 Use of Color (Level A) · MODERATE**

> **Fixed (fbc97dd+):** footer "About" and "Source code" links now carry
> `text-decoration:underline` in `templates/base.html`, giving a non-colour cue.

Footer links ("About", "Source code") render as
`style="color:#3DB4E0; text-decoration:none;"` — no underline, no non-colour
cue, so they're only tellable from surrounding text by hue. axe confirmed
(`link-in-text-block`). The same risk applies to any inline link that drops the
underline.

- **Fix (low):** underline inline text links (or add another non-colour
  affordance). Standalone "button-like" links and nav items are exempt — this is
  specifically about links sitting *inside a block of text*.

### 🟡 F5 — No reduced-motion support
**WCAG 2.3.3 Animation from Interactions (AAA) + best practice · MODERATE**

There is **no `prefers-reduced-motion` rule anywhere** in the CSS. The HTMX
loading bar animates on every request, and hover/transition effects run
unconditionally. 2.3.3 is AAA, but honouring the OS "reduce motion" setting is a
baseline expectation for a government tool and helps users with vestibular
disorders.

- **Fix (low):** add a global
  `@media (prefers-reduced-motion: reduce) { *, *::before, *::after { animation-duration: 0.01ms !important; transition-duration: 0.01ms !important; } }`
  override, or scope it to the loading bar + transitions.

### 🟡 F6 — Faint UI-component borders may fail non-text contrast
**WCAG 1.4.11 Non-text Contrast (Level AA) · MODERATE · needs confirmation**

`--color-border: rgba(100,140,180,0.07)` composites to roughly **~1.1:1** against
the base surface — far below the 3:1 that 1.4.11 requires for the visual boundary
of a control (input, select, card edge). Whether this *fails* depends on whether
the border is the **only** thing identifying the control; some inputs also carry
a subtle fill. Flagged here because the number is so low it almost certainly
needs attention.

- **Fix (low):** raise the border alpha for *interactive* components
  (inputs/selects/buttons) to clear 3:1, independent of the decorative card-edge
  border. A good candidate to settle during the Phase F depth/border pass.

### ✅ F7 — Feedback widget: unlabeled file input — RESOLVED
**WCAG 4.1.2 Name, Role, Value (Level A) · MINOR**

> **Fixed (fbc97dd+):** the file input now carries
> `aria-label="Attach screenshot images"` in
> `templates/partials/_feedback_widget.html`.

The hidden file input in the feedback widget
(`<input type="file" id="oh2o-fb-file" class="oh2o-fb-hp" tabindex="-1">`) has no
accessible name (axe `label`). It's `tabindex="-1"`, so low impact, but trivially
fixed.

- **Fix (trivial):** add `aria-label="Attach a screenshot"` to the input.
- **Where:** `templates/partials/_feedback_widget.html`.

### ⚪ F8 — Feedback modal heading order
**WCAG 1.3.1 (A) · MINOR · tied to F3**

The feedback modal's `<h3 id="oh2o-fb-title">` is the only other heading on most
pages, producing an `H1 → H3` jump (axe `heading-order`). Largely a symptom of
F3 (no real page `<h2>`s between them). Re-check after F3 is fixed; if the modal
still jumps, make it an `<h2>` or give the dialog its own heading context.

## Staging-only — not real defects

Lighthouse "best practices" flagged **HTTPS / redirect-to-HTTPS / a
Cross-Origin-Opener-Policy console error**. All three are because the *staging*
box serves plain HTTP over Tailscale. **Production runs behind Caddy with
auto-HTTPS**, so none of these apply to the shipped product. No action.

## Recommended additional manual verification

Things this pass did not fully exercise and that deserve a hands-on check before
declaring AA conformance:

- **Feedback modal focus management** — when it opens, does focus move into the
  dialog, stay trapped while open, and return on close? Does Escape close it?
  (2.4.3 / 2.1.2.)
- **HTMX live-region announcements** — when a master-detail row swaps the detail
  pane, is the change announced to a screen reader? Consider an `aria-live`
  region or focus move to the new pane heading. (4.1.3.)
- **Full form-label sweep** — every create/edit page (the audit spot-checked the
  ledger filters and found them clean, but did not open every form).
- **One real screen-reader pass** (VoiceOver) end-to-end on each bucket.

## Suggested remediation order

1. **F1 (Critical, low effort)** — keyboard activation for master rows. Ship this
   first; it's the one true Level-A blocker and it's tiny.
2. **F3 + F8 (Serious, low–med)** — fix the `<h1>`/heading structure (also closes
   a real template bug).
3. **F4, F6, F7 (Moderate/Minor, low)** — link underlines, interactive-border
   contrast, the widget label. Small, independent.
4. **F2 + F5 (Serious + Moderate)** — fold the contrast-token changes and the
   reduced-motion media query into **Phase F**, since both are theme-level and
   Phase F is already going to touch colour, depth, and motion. Doing them there
   means the visual pass is done once on final tokens.

This sequence clears the Level-A blocker immediately, knocks out the cheap wins,
and lets the theme-level items ride the aesthetics phase they naturally belong to.
