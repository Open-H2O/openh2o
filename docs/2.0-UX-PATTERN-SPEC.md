**Why this exists:** The 2.0 work was scoped as "convert screens to master-detail."
That was wrong. Master-detail is one of *three* layout patterns, and which one a
screen should use depends on the user's task, the number of records, and how heavy
each record's detail is — not on a template applied uniformly. Several screens were
converted to master-detail that should not have been. This document defines the
deciding rule, assigns every screen to a pattern, lists the cross-cutting UX work
2.0 needs beyond layout, and fixes the sequence — ending with the purely-aesthetic
Apple overhaul as the final gate before the v2.0 tag.

---

## The core principle

**Layout follows the user's task, not the record count.** Count and detail-weight
are the two physical constraints that then decide which layout can actually serve
that task.

The three questions a screen answers map to the three layouts:

- "Which one, and what's its story?" → **master-detail**
- "How much, in total?" → **data table**
- "Show me this one thing, fully." → **overview + full detail page**

**Persona lens (decisive for this app):** users are non-technical staff at small,
under-resourced agencies who use the tool *infrequently* (seasonally, around
reporting deadlines). The governing principle for that audience — per USWDS, the
GOV.UK Design System, and Nielsen Norman Group — is **recognition over recall**:
the screen must re-teach itself every visit through plain language, visible options,
and forgiving findability. This is why the cross-cutting work (search, inline help)
matters as much as layout.

## The three buckets

### Bucket 1 — Master-detail workspace
**For:** many items, each with a medium-rich story; task is compare-and-browse with
frequent switching while staying in context.

**UI:** persistent searchable list (left) + selected item's detail (right), with a
map at the top of the detail when the item is spatial. Clicking a row swaps the
detail in place; the list never moves. A heavy detail offers an "open full page"
escape.

**Responsive collapse:** below ~720px the two columns can't coexist — collapse to
Bucket 3 (list, then a full detail view). This is a requirement, not an option.

**Screens:** Use Areas, Wells, Water Accounts.

### Bucket 2 — Dense data table
**For:** many items treated as *data*, thin per-row detail; task is filter, sort,
compare columns, and read totals.

**UI:** the whole screen is one table — filters/search across the top, sortable
columns, sticky header, and column **totals/subtotals** at the bottom. No detail
pane (there is nothing rich enough to warrant one); a row may link out to a related
record. Preset filters ("This Period," "Active Use Areas") aid infrequent users.

**Responsive collapse:** prioritize the key columns; allow horizontal scroll with a
frozen identifier column.

**Screens:** Use Ledger. (Allocations likely belongs here too — confirm in Phase D.)

### Bucket 3 — Overview, then full detail page
**For:** few items, each heavy; task is read-one-deeply, or triage then dive.

**UI:** two screens. **Overview** = a wide map on top showing all items at once,
with the list as full-width rows/cards below — no cramped side column. Clicking an
item (map point or list row) navigates to its **own full-width detail page**, where
the big map, charts, and sub-tables have room. Back returns to the overview. This is
the Monitoring Stations model.

**Responsive collapse:** naturally mobile-friendly — map stacks above list; detail
is already its own page.

**Non-spatial exception:** the overview map applies only when the item has
geometry of its own. **Water Rights** is Bucket 3 by count/weight but is a legal
entitlement with no geometry — so its overview is the full-width list alone, and
its spatial footprint (the diversion points it authorizes) appears on the detail
page, not the overview.

**Screens:** Zones, Recharge, Water Rights, Surface Diversions, Monitoring Stations.

## Cross-cutting 2.0 upgrades (beyond layout)

Per-screen layout work does not touch these, and they are what make the app "easy to
use" for an infrequent, non-technical user.

**Tier 1 — foundational:**
- **Global search.** Today search is per-screen; there is no way to find "parcel
  MER-APN-014" without first knowing it lives under Use Areas. A top-bar search
  across parcels, wells, accounts, zones, and stations is the single highest-value
  addition in 2.0.
- **Bucket consistency.** Finish the system: re-bucket the three live screens, move
  Use Areas onto the shared shell, make the Ledger a real table. Consistency is a
  usability feature — it lets a returning user's skill transfer.
- **Responsive collapse** defined per bucket (above).

**Tier 2 — ease of use & flow:**
- **Inline contextual help.** A real glossary exists but is buried under Help. Add
  "?" info icons at the point of confusion and inline-link glossary terms.
- **Dashboard "what needs attention" glance.** The dashboard is tutorial-first
  (good for a fresh instance, thin for a returning admin). Add live exception tiles
  (stations down, periods unclosed, allocations over budget) above the tutorial.
- **Unify onboarding.** New users meet both a Setup Wizard and empty-state step
  lists; pick one spine.

**Tier 3 — compliance & polish:**
- **Accessibility pass** — add a skip-to-content link, run a formal WCAG 2.1 /
  Section 508 check. Required for a government tool; also helps older/low-confidence
  users.
- **Consistent visual grammar** — folded into the aesthetics pass below.

### Phase E sub-phases (scoped against the code, 2026-06-22)

Six cross-cutting items, ordered to lead with a fast self-contained win, then the
marquee feature, then polish. Each is its own sub-phase so we stay in the loop
between them. Build order ≠ spec tier order: a cheap high-value tile beats a big
feature for momentum. **Current state below is what the code actually has today —
three items are already part-built, so the real Phase E is smaller than it looks.**

- ✅ **E1 — Dashboard "what needs attention" strip (DONE 2026-06-22).** Brent
  picked the **compact strip** over prominent cards. A single-row exception bar at
  the top of the dashboard content (`_attention_strip.html`), each counter a
  clickable pill rendered only when non-zero; collapses to one "All clear" line
  when every count is zero. Counts: **periods to close** (`is_finalized=False` +
  `end_date` past → periods list), **stations down** (active stations classed
  `dead` by `datasync.freshness`, source-aware so it matches the Monitoring screen
  → station list), **accounts over budget** (active accounts whose consumptive use
  passed allocation this period → jumps to `#active-accounts`). Lives inside the
  HTMX-swapped content so the period-dependent over-budget count refreshes on
  period change. Verified on staging (1 / 3 / 5 live counts; browser + computed
  styles); +3 tests, full suite 882 passed. CSS in app.css (v51→v52).
- ⬜ **E2 — Global search** (marquee Tier-1 feature). *Current:* NONE — the top
  header (`_header.html`) has no search slot; every search is per-screen. *Work:* a
  search view spanning parcels / wells / accounts / zones / stations; a top-bar
  HTMX input with a live results dropdown; route each hit to its screen. The single
  highest-value addition, but the biggest build + most design surface (what to
  search, ranking, result UI). *Medium–large; a multi-sitting effort on its own.*
- ✅ **E3 — Accessibility quick wins (DONE 2026-06-22).** Added a skip-to-content
  link as the first focusable element in `base.html` (off-screen until focused,

  Verified on staging (focus reveals it at top:16px, gold-on-dark, jumps to main).

**Open decisions for Brent (before the gated sub-phases):** E2 — scope of global
search (which entities, how results route); E6 — which onboarding spine wins.

### Phase F — Apple aesthetics overhaul (the very end)

**This is the last step before 2.0 ships, and it is purely aesthetic** — type,
spacing, depth, color refinement, and motion, so the web app reads like the native
Mac app. It changes *no* structure, layout pattern, or behavior.

It comes last **by design**: structure stops moving first, so the visual polish is
applied once to settled screens and never redone as layouts change.

**v2.0 does not ship until Phase F is complete.** The intermediate screen work ships
under incrementing point versions (v1.x); the `v2.0` tag is the finish line, applied
only after the aesthetics overhaul lands on top of finished structure.

## Research backing

The three-bucket logic matches established guidance: Nielsen Norman Group (data
tables; list entries; complex application design; progressive disclosure),
Microsoft's master-detail guidance (incl. the ~720px split-view threshold), and
enterprise data-table UX literature. The persona principles (task-based navigation,
plain language, recognition over recall, findability, accessibility) come from the
US Web Design System, the GOV.UK Design System, and NN/g's government/accessibility
guidance.
